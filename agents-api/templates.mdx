title: "Templates"

A template is a prebuilt image that decides which agents an instance installs and which agents the gateway can route sessions to. Pass `template` (a template name) when you create an [instance](/agents-api/instances), or omit it for the default. Direct image references are not accepted on instance create; register a custom template first, then pass its name.

## Built-in templates

Built-in templates are reserved under the `agent37-` prefix.

| Template | Installs | Availability |
| --- | --- | --- |
| `agent37-hermes` | Hermes, general agent: browser, code, files. | Default, available today |
| `agent37-openclaw` | OpenClaw gateway agent. | Available today |
| `agent37-claude-code` | Claude Code, for software work. | Coming soon |
| `agent37-codex` | Codex, for software work. | Coming soon |
| `agent37-all` | Hermes, Claude Code, Codex, and OpenClaw together. | Coming soon |

```bash curl
curl -X POST https://api.agent37.com/v1/instances \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{ "template": "agent37-hermes" }'
```

Omit `template` and you get `agent37-hermes`.

<Note>
  **Multi-agent templates.** When a template installs more than one agent (like `agent37-all`), the gateway routes a new session to whichever agent you choose with the `agent` field on its [first message](/agents-api/messages). The choice is fixed for the session's life. Single-agent templates ignore it.
</Note>

## Custom templates

<Info>
  Custom templates are in preview, and for now they start from a **public** image you host. Build an image `FROM` our base, push it to a public registry you already use, then register it with us and create instances from it. Private images and pushing straight to our own registry are coming soon.
</Info>

A custom template is your own image, built on our base and registered under a name you choose. Use it to preinstall [skills, plugins, and packages](/agents-api/skills#preinstall-with-a-custom-image) so every instance you create from it starts ready. Custom templates are scoped to your workspace and are not visible to other workspaces.

<Steps>
  <Step title="Build on our base">
    Write a Dockerfile that starts `FROM ghcr.io/agent37/base:latest` and layer your skills, plugins, and packages on top. The runtime, gateway, terminal, desktop, file browser, and managed auth all come from the base and keep working; the container has passwordless `sudo`, so you can install anything. See [Skills & plugins](/agents-api/skills#preinstall-with-a-custom-image) for the full Dockerfile.

    ```bash
    docker build --platform=linux/amd64 -t my-agent:v1 .
    ```

    Build for `linux/amd64` even on an Apple Silicon Mac. Instances run on amd64, and an arm64 image will fail to start.
  </Step>

  <Step title="Push it to a public registry">
    Push the image to any registry you already use, such as Docker Hub or GitHub Container Registry, and make sure it is public so we can pull it. You log in to your own registry the usual way; we never see your registry password.

    ```bash
    docker tag my-agent:v1 ghcr.io/your-name/my-agent:v1
    docker push ghcr.io/your-name/my-agent:v1
    ```
  </Step>

  <Step title="Register it with us">
    Tell us the image to pull and a name to call it. We check that `image_ref` is a fully qualified public reference and store the template as `active` right away. We pull the image the first time you create or update an instance from it, so a bad reference surfaces as a failed instance create, not here.

    ```bash curl
    curl -X POST https://api.agent37.com/v1/templates \
      -H "Authorization: Bearer sk_live_..." \
      -H "Content-Type: application/json" \
      -d '{ "name": "my-agent", "image_ref": "ghcr.io/your-name/my-agent:v1", "description": "My agent" }'
    ```

    The `name` is 2 to 63 characters, lowercase letters, numbers, and hyphens, starting with a letter. The `agent37-` prefix is reserved for built-in templates. Pin a tag on `image_ref`; do not rely on `latest`.
  </Step>

  <Step title="Create instances from it">
    Pass the name you registered as `template` when you create an [instance](/agents-api/instances).

    ```bash curl
    curl -X POST https://api.agent37.com/v1/instances \
      -H "Authorization: Bearer sk_live_..." \
      -H "Content-Type: application/json" \
      -d '{ "template": "my-agent" }'
    ```
  </Step>
</Steps>

### Manage templates

| Action | Endpoint |
| --- | --- |
| List built-in and workspace templates | `GET /v1/templates` |
| Get one | `GET /v1/templates/{name}` |
| Register an image you host | `POST /v1/templates` |
| Update name, image, description, or status | `PATCH /v1/templates/{name}` |
| Deactivate a workspace template | `DELETE /v1/templates/{name}` |

Built-in (`agent37-`) templates are read-only: `PATCH` and `DELETE` on them return `403`.

`PATCH` changes any of `name`, `image_ref`, `description`, or `status` (`active` or `inactive`). Point `image_ref` at an older tag to roll back, or set `status` to `inactive` to stop new instances from resolving the template.

`DELETE` deactivates the template (sets `status` to `inactive`) and acts once: the first call returns `200` with `{ "name": "my-agent", "deleted": true }`, and a repeat or already-inactive template returns `404`. Existing instances created from it keep running on the image they already pulled.

<ResponseField name="name" type="string">
  The name you pass as `template` when creating an instance, e.g. `my-agent`.
</ResponseField>
<ResponseField name="scope" type="string">
  `system` for built-in `agent37-` templates, `workspace` for ones you register.
</ResponseField>
<ResponseField name="image_ref" type="string">
  The image we pull, e.g. `ghcr.io/your-name/my-agent:v1`. Pin a tag; do not rely on `latest`.
</ResponseField>
<ResponseField name="agents" type="string[]">
  The agents the template installs.
</ResponseField>
<ResponseField name="status" type="string">
  `active` (resolvable on create) or `inactive` (deactivated, or a built-in that is coming soon).
</ResponseField>
<ResponseField name="description" type="string">
  Your optional description, or `null`.
</ResponseField>
<ResponseField name="created" type="integer">
  Creation time in unix seconds, or `null` for built-in templates.
</ResponseField>
<ResponseField name="updated" type="integer">
  Last update time in unix seconds, or `null` for built-in templates.
</ResponseField>

<Tip>
  Building `FROM` our base is what gives an instance the managed agent, chat, terminal, desktop, and files. An image that is not built from our base still runs, but as a plain container without them.
</Tip>

<Note>
  **Coming soon.** Registering a **private** image (you give us a read-only pull token for your registry) and pushing an image **straight to our registry** at `registry.agent37.com`, so you can skip hosting it yourself. For now, use a public image.
</Note>
