title: "Build a custom image"
description: "Build a linux/amd64 Docker image in the cloud or import it from a public registry, and run instances from the pinned template."
---

You can run an Agent37 instance from any Docker image. Publish it as a [workspace template](/agents-api/templates) in one of two ways:

| Your image | Publish with | What Agent37 needs |
| --- | --- | --- |
| A Dockerfile, anywhere | A [cloud build](#build-in-the-cloud) | Run `npx agent37 templates build` in its folder. Agent37 builds and publishes it — no local Docker, and nothing has to be public. |
| Pre-built, in a public registry | `image_ref` | A fully qualified reference that Agent37 can pull anonymously during registration. |

Neither path accepts private-registry credentials. Cloud builds pull `FROM` images from public registries only, and a pre-built image can be imported only from a public registry — an image that exists only in a private registry has to be rebuilt from its Dockerfile (on a public base) or republished under a public tag.

A complete example — a `Dockerfile`, an example skill, and a tiny LLM proxy — lives in [`custom-agent-image/`](https://github.com/agent37-platform/starter-kit/tree/main/examples/custom-agent-image), ready to copy into your own repo.

This page is the walkthrough; [Templates → build on the Hermes base image](/agents-api/templates#build-on-the-hermes-base-image) is the reference for the contract.

<Note>
  When you publish a template, Agent37 copies your image once into private storage and pins it by digest. That snapshot is what every instance runs, so publishing takes up to a few minutes for a large image. Re-pushing the same registry tag later does nothing: publish a new tag and [update the template](/agents-api/templates#update-a-template), or re-run the cloud build. Your image is stored privately and never republished.
</Note>

## 1. Choose a starting image

To customize Hermes, build on [`ghcr.io/agent37-platform/hermes-base`](https://github.com/orgs/agent37-platform/packages/container/package/hermes-base). It includes Hermes, the gateway, Chromium, and the standard toolchain. This example adds a CLI and a skill:

```dockerfile
FROM ghcr.io/agent37-platform/hermes-base:latest

USER root
RUN apt-get update && apt-get install -y --no-install-recommends your-cli \
 && rm -rf /var/lib/apt/lists/*
COPY your-skill/ /usr/local/share/agent37/default-skills/your-skill/
USER node
```

Bake binaries into `/usr/local/bin`, skills into `/usr/local/share/agent37/default-skills/` (the entrypoint copies them to `~/.hermes/skills` at boot), and everything else into `/usr/local` or `/opt` — never `/home/node` or `/home/linuxbrew`, which are masked at runtime. Keep the base `ENTRYPOINT`. See [the full contract](/agents-api/templates#build-on-the-hermes-base-image).

You can also use an existing image or start from any base. Its main process must keep running. If it serves HTTP, bind it to `0.0.0.0` and note its listening port; you will pass that as `default_port` when you register the template. An image with no HTTP service can omit `default_port` and be driven through [exec](/agents-api/exec), but it still needs a long-running `ENTRYPOINT` or `CMD`.

<Tip>
  Your built image freezes its base at build time. `:latest` is convenient while getting started; pin a dated Hermes base tag for reproducible production rebuilds. Find published tags [on GHCR](https://github.com/orgs/agent37-platform/packages/container/package/hermes-base), or read the current tag from `agent37-hermes`'s `image_ref` on `GET /v1/templates`.
</Tip>

## 2. Verify it locally (optional)

If you have Docker, build and run the image locally before publishing — publication validates the image format and architecture, but only a running container proves that its entrypoint and service work. `--load` makes the amd64 result runnable, including on an Apple Silicon Mac:

```bash
docker buildx build --platform linux/amd64 --load -t my-agent:v1 .
docker image inspect my-agent:v1 --format '{{.Os}}/{{.Architecture}}'
# linux/amd64
```

No local Docker? Skip this — the cloud build below builds for you, and the streamed build log plus a test instance play the same role.

## 3. Publish the image

You have two ways to get the image to Agent37. Either way it ends up **`linux/amd64`**, and the platform copies it once into private storage.

<Note>
  Importing a registry image is synchronous and can take a few minutes for a large image; a cloud build is asynchronous — poll it (the CLI does) until it succeeds. Either way, the resulting `image_digest` identifies the immutable private copy every instance runs. Re-pushing a mutable source tag does not change that copy.
</Note>

### Build in the cloud

The path for anything local or private: run the Agent37 CLI in the folder with your
Dockerfile. It packs the folder as the build context, Agent37 builds the image on its own
infrastructure (streaming the build log back live), and the finished image is published as
your template — created at revision 1, or a new revision if the name already exists:

```bash
export AGENT37_API_KEY=sk_live_...
npx agent37 templates build --name my-custom-agent
```

Pass a directory to build somewhere else (`npx agent37 templates build ./agent-image`); omit
`--name` to use the folder's name. Add `--default-port 8000` if your image serves HTTP on a
port (see the tip below). The CLI reads only `AGENT37_API_KEY`; there is no CLI login or
stored session. Ctrl-C detaches the log but never cancels the build — it finishes
server-side. Existing instances are never updated by this command.

<Warning>
  The **whole folder** ships as the build context, not just the Dockerfile — minus `.git` and
  plain `.dockerignore` patterns. A stray `.env` or credentials file in the folder travels
  with the context, and a `COPY . .` bakes it into the image; add a `.dockerignore` first.
  The packed context must stay under 100 MB, so keep it to the Dockerfile and the files it
  COPYs.
</Warning>

Because the build environment holds no registry credentials, every `FROM` in the Dockerfile
must be publicly pullable (the `agent37-platform` base images are). Builds are free, run one
at a time per workspace, and support images up to about 4 GB today. The full contract and the
raw HTTP API are in [Templates → build a template in the cloud](/agents-api/templates#build-a-template-in-the-cloud).

### Import a public registry image

Push the image to any public OCI registry, then register its fully qualified reference. For example:

```bash
docker tag my-agent:v1 ghcr.io/you/my-agent:v1
docker push ghcr.io/you/my-agent:v1
```

On GHCR, make the package public after the first push (**Packages** → **Package settings** → **Change visibility** → **Public**). The [starter example](https://github.com/agent37-platform/starter-kit/tree/main/examples/custom-agent-image) includes a GitHub Actions workflow that builds `linux/amd64` and publishes immutable commit tags.

Register a specific tag, not `latest`:

```bash curl
curl -X POST https://api.agent37.com/v1/templates \
  -H "Authorization: Bearer $AGENT37_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "name": "my-custom-agent", "image_ref": "ghcr.io/you/my-agent:v1" }'
```

Agent37 needs anonymous access only while it copies the image. The template and its instances no longer depend on the source registry after registration. A future image update needs a new public tag, or a fresh cloud build.

<Tip>
  If your own image serves HTTP on port `8000`, declare it: `--default-port 8000` on the build, or `"default_port": 8000` beside `name` and `image_ref` on register. The bare instance URL then routes to that port, and instance creation probes it at boot. Omit it for `hermes-base`, whose gateway uses the `3737` fallback, or for a long-running private sandbox with no HTTP service.
</Tip>

## 4. Create an instance

Register once, then create instances from the template name:

```bash curl
curl -X POST https://api.agent37.com/v1/instances \
  -H "Authorization: Bearer $AGENT37_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "template": "my-custom-agent" }'
```

The result is a standard Agent37 instance running your image: same [lifecycle](/agents-api/instances), [exec](/agents-api/exec), and routed URLs as any other. Confirm your CLI shipped, without needing a model:

```bash curl
curl -X POST https://api.agent37.com/v1/instances/<id>/exec \
  -H "Authorization: Bearer $AGENT37_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "command": "your-cli --version" }'
```

<Note>
  If the instance comes up `failed` instead of `running`, read its [logs](/agents-api/logs). An entrypoint that exits, a missing binary, or a service that never listens on `default_port` are the usual causes. `exec` cannot help until the container is running.
</Note>

## 5. Bring your own model

`hermes-base` is clean: it boots with no LLM provider (standard Agent37 instances use Agent37's managed model; this base is for bringing your own). To run an instance on your own model, point Hermes at any OpenAI-compatible endpoint — your own proxy, or a provider directly — by writing `~/.hermes/config.yaml` on the instance:

```yaml
model:
  provider: "custom:MyProvider"
  default: "moonshotai/kimi-k2.7-code"        # the model id your endpoint serves
custom_providers:
  - name: "MyProvider"
    base_url: "https://your-llm-proxy.example.com/v1"   # must end in /v1
    api_key: "your-proxy-token"
    api_mode: "chat_completions"
    model: "moonshotai/kimi-k2.7-code"
```

Your endpoint must serve the two OpenAI-compatible routes Hermes uses: `GET /v1/models` (to resolve the model id) and `POST /v1/chat/completions` (the turn). The example folder includes a ~40-line `llm-proxy/` that implements exactly these and forwards to OpenRouter with your key — a minimal pattern to deploy and adapt.

Write the config over [exec](/agents-api/exec) or the instance terminal; it lives on the persistent volume, so it survives restarts. Then [send a message](/agents-api/chat) and the agent runs on your model.

<Tip>
  Want chat to work out of the box on Agent37's managed model instead? Build `FROM ghcr.io/agent37-platform/hermes:<tag>` ([tags on GHCR](https://github.com/orgs/agent37-platform/packages/container/package/hermes)) — the full image wires the managed model — and pass a [budget](/agents-api/budgets) on create.
</Tip>

## Keep it current

Your image freezes its base; rebuilding is how it picks up platform updates. Each template revision is an immutable snapshot, so a new build takes effect only when the template's image changes: re-run `npx agent37 templates build`, or publish a **new registry tag** and PATCH `image_ref` ([update the template](/agents-api/templates#update-a-template)). A changed `image_ref` or every finished cloud build increments the template's automatic `revision`; a build counts even when its digest matches the previous image. PATCHing the same `image_ref` string deliberately reuses the existing snapshot and revision.

Existing instances keep their installed `template_revision`. [Update each instance](/agents-api/instances#update) to recreate it from the template's current revision. A skill already seeded into an instance's `~/.hermes/skills` is not overwritten on update — only fresh instances get a changed skill.

## Troubleshooting

| Error or symptom | What to check |
| --- | --- |
| Build `status` is `failed` with `build_failed` | A Dockerfile step failed on the cloud builder. The streamed log shows the failing step; fix it and re-run — a `FROM` that is not publicly pullable fails here too. |
| `build_timeout` | The build ran over the 45-minute limit. Trim slow steps, or split work out of the Dockerfile. |
| `build_conflict` | The build was already started, or another build is running in your workspace. Wait for it (one build at a time). |
| `invalid_image_archive` | The image is not a single `linux/amd64` image. Rebuild for `linux/amd64`. |
| `image_too_large` | The staged image is over the cap. Remove build caches and unnecessary layers, or contact support. |
| `invalid_request` for `image_ref` | Use a fully qualified tag and confirm the registry allows anonymous pulls and the tag has a `linux/amd64` image. A private image can't be imported: rebuild it from its Dockerfile with a [cloud build](#build-in-the-cloud), or republish it under a public tag. |
| `image_ingest_failed` | The private copy failed on Agent37's side. Retry once, then contact `support@agent37.com`. |
| Instance status is `failed` | Read [instance logs](/agents-api/logs); confirm the main process stays alive and `default_port` listens when set. |
