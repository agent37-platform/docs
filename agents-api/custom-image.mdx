description: "Start from the Hermes base, add your tools and skills, register your own image from a public registry or a local upload, and run instances from it — optionally on your own model."
---

You can run instances on your own Docker image: start from the Hermes base, add the CLIs, skills, and config your agent needs, and [register it as a template](/agents-api/templates) — either from a **public registry** or by **uploading a local image**, no registry required. You can also point the agent at your own model. A complete, copy-able example — a `Dockerfile`, an example skill, a `register.sh`, and a tiny LLM proxy — lives in [`custom-agent-image/`](https://github.com/agent37-platform/starter-kit/tree/main/examples/custom-agent-image), ready to copy into your own repo.

This page is the walkthrough; [Templates → build on the Hermes base image](/agents-api/templates#build-on-the-hermes-base-image) is the reference for the contract.

<Note>
  When you register a template, Agent37 copies your image once into private storage and pins it by digest. That snapshot is what every instance runs, so registering takes up to a few minutes for a large image, and re-pushing the same tag later does nothing until you [update the template](/agents-api/templates#update-a-template). Your image is stored privately and never republished.
</Note>

## 1. Start from the base

Your `Dockerfile` builds on [`ghcr.io/agent37-platform/hermes-base`](https://github.com/orgs/agent37-platform/packages/container/package/hermes-base) and adds your layers. This example installs a CLI into `/usr/local/bin` and a skill into the default-skills directory:

```dockerfile
FROM ghcr.io/agent37-platform/hermes-base:latest

USER root
RUN apt-get update && apt-get install -y --no-install-recommends your-cli \
 && rm -rf /var/lib/apt/lists/*
COPY your-skill/ /usr/local/share/agent37/default-skills/your-skill/
USER node
```

Bake binaries into `/usr/local/bin`, skills into `/usr/local/share/agent37/default-skills/` (the entrypoint copies them to `~/.hermes/skills` at boot), and everything else into `/usr/local` or `/opt` — never `/home/node` or `/home/linuxbrew`, which are masked at runtime. Keep the base `ENTRYPOINT`. See [the contract](/agents-api/templates#build-on-the-hermes-base-image).

<Tip>
  Either way, the image you push freezes its base at build time — the tag choice only matters when you rebuild. `:latest` tracks the newest base, so getting-started builds never go stale. For reproducible production rebuilds, pin a date tag instead: the tag list is [on GHCR](https://github.com/orgs/agent37-platform/packages/container/package/hermes-base), and the current one is the tag in `agent37-hermes`'s `image_ref` on `GET /v1/templates`.
</Tip>

Test the build before you publish:

```bash
docker build --platform linux/amd64 -t my-agent .
```

## 2. Register the image

You have two ways to get the image to Agent37. Either way it must be built for **`linux/amd64`** and be **at most 5 GB**, and either way the platform copies it once into private storage at registration.

<Note>
  Build for `linux/amd64` even on an Apple Silicon Mac (`docker build --platform=linux/amd64 ...`). A stray arm64 image is rejected at registration.
</Note>

### From a public registry

Push the image to any public registry, then register it by reference. GitHub Container Registry is the least setup, because GitHub Actions can build and push it with the built-in token — no secrets. The example folder ships a reference workflow that does exactly this. On GHCR, make the package public once after the first push (**Packages** → **Package settings** → **Change visibility** → **Public**) so Agent37 can pull it to copy it.

Register the reference as a template (pin a tag, not `latest`):

```bash curl
curl -X POST https://api.agent37.com/v1/templates \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{ "name": "my-custom-agent", "image_ref": "ghcr.io/you/my-agent:<tag>" }'
```

### From a local image (no registry)

If your image is private or you never published it, upload the `docker save` archive directly. Ask for an upload slot, `PUT` the archive to the returned URL, then create the template from the `upload_id` — no registry, no public exposure.

```bash curl
# Save the built image to a tarball (gzip optional).
docker save my-agent:<tag> -o my-agent.tar

# 1. Ask for an upload URL.
curl -X POST https://api.agent37.com/v1/template-uploads \
  -H "Authorization: Bearer sk_live_..."
# -> { "id": "up_...", "upload_url": "https://...", "expires_at": 1752... }

# 2. Upload the archive to the returned upload_url.
curl -T my-agent.tar "<upload_url>"

# 3. Register the template from the upload.
curl -X POST https://api.agent37.com/v1/templates \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{ "name": "my-custom-agent", "upload_id": "up_..." }'
```

An upload-born template exposes only its `image_digest` — there is no `image_ref`, because there is no registry the image lives in.

## 3. Create an instance

Register once, then create instances from the template name:

```bash curl
curl -X POST https://api.agent37.com/v1/instances \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{ "template": "my-custom-agent" }'
```

The result is a standard Agent37 instance running your image: same [lifecycle](/agents-api/instances), [exec](/agents-api/exec), and routed URLs as any other. Confirm your CLI shipped, without needing a model:

```bash curl
curl -X POST https://api.agent37.com/v1/instances/<id>/exec \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{ "command": "your-cli --version" }'
```

<Note>
  If the instance comes up `failed` instead of `running`, your image did not boot. Read its [logs](/agents-api/logs) for the entrypoint's own error. A missing binary, a bad path, or a wrong-architecture build are the usual causes, and `exec` cannot help here since there is no running container to attach to.
</Note>

## 4. Bring your own model

`hermes-base` is clean: it boots with no LLM provider (standard Agent37 instances use Agent37's managed model; this base is for bringing your own). To run an instance on your own model, point Hermes at any OpenAI-compatible endpoint — your own proxy, or a provider directly — by writing `~/.hermes/config.yaml` on the instance:

```yaml
model:
  provider: "custom:MyProvider"
  default: "moonshotai/kimi-k2.7-code"        # the model id your endpoint serves
custom_providers:
  - name: "MyProvider"
    base_url: "https://your-llm-proxy.example.com/v1"   # must end in /v1
    api_key: "your-proxy-token"
    api_mode: "chat_completions"
    model: "moonshotai/kimi-k2.7-code"
```

Your endpoint must serve the two OpenAI-compatible routes Hermes uses: `GET /v1/models` (to resolve the model id) and `POST /v1/chat/completions` (the turn). The example folder includes a ~40-line `llm-proxy/` that implements exactly these and forwards to OpenRouter with your key — a minimal pattern to deploy and adapt.

Write the config over [exec](/agents-api/exec) or the instance terminal; it lives on the persistent volume, so it survives restarts. Then [send a message](/agents-api/chat) and the agent runs on your model.

<Tip>
  Want chat to work out of the box on Agent37's managed model instead? Build `FROM ghcr.io/agent37-platform/hermes:<tag>` ([tags on GHCR](https://github.com/orgs/agent37-platform/packages/container/package/hermes)) — the full image wires the managed model — and pass a [budget](/agents-api/budgets) on create.
</Tip>

## Keep it current

Your published image freezes its base; rebuilding is how it picks up platform updates. Because the template is an immutable snapshot, a new build only takes effect once you re-register it: rebuild, then either point the template at the new tag (`PATCH /v1/templates/{name}` with `image_ref`) or push a fresh upload and `PATCH` the new `upload_id`. Re-pushing the same registry tag alone changes nothing. Then [update each instance](/agents-api/instances) to pick it up. A skill already seeded into an instance's `~/.hermes/skills` is not overwritten on update — only fresh instances get a changed skill.
